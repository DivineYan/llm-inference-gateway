# M3 —— Agent 编排层概要设计（HLD）

> 上游文档：《AI 推理调度平台 —— 产品需求文档 v1.1》
> 本阶段范围：Layer 2 Agent 编排层
> 对应需求：FR-2.1 任务编排 / FR-2.2 工具调用（mini MCP） / FR-2.3 任务状态管理（幂等/断点续跑）
> 核心难点：TD-4 Agent 任务幂等
> 文档性质：概要设计（HLD）。不含代码，但详细到可直接对照开工。

---

## 0. 本阶段目标

**部署目的（先钉死）**：把编排层落地为一个**平台内置的 AIOps Copilot**——对本推理平台自身的运行做

1. **根因诊断**（探索式，回答"为什么错/慢"）；
2. **运营报表**（确定式，定期产出用量/SLA/成本）；
3. **优化建议**（只读，产出调权/调限流等建议，不自动改）。

M3 **全程只读**：agent 只观测与建议，不写任何配置（自动修复 = Tier 4，留作后续，见 §10）。

**两条原则贯穿设计**：
- **dogfood 起步、留外部适配口**：M3 分析的是 M1/M2 已产出的本平台 telemetry；工具抽象预留"接外部 MCP server"的位置（见 §4.2）。
- **model-agnostic**：模型后端是可替换适配器，mock 与真实模型同契约，接真模型只换适配器、编排零改（见 §4.1）。

M3 结束时应能演示：

1. 注入一个故障（某后端熔断）→ **ReAct agent 调工具逐步定位到根因**（如"gpt-a 5xx 连续失败触发熔断，503 由 served_fallback 兜底"）。
2. 触发「用量/SLA 周报」**Workflow** → 取数→分析→出报表；中途失败重跑不重复已完成步骤。
3. 喂入"某后端失败率高"的 telemetry → **优化建议**产出"建议给 gpt-a 降权"。
4. 所有模型调用复用 M2 保障层；查询任务状态看到状态机与中间结果。

## 1. 编排层在系统中的位置

```
   调用方 ──► [M1 网关] ──► [M2 保障] ──► 模型后端
                  │            ▲
                  │ 产出 telemetry │ 复用（每次模型调用都走保障层）
                  ▼            │
        ┌─────────────────────┴───────────────────┐
        │           M3 AIOps Copilot（编排层）       │
        │  读本平台 telemetry → 诊断 / 报表 / 建议     │
        └──────────────────────────────────────────┘
```

编排层是 M1/M2 的**消费者**：它读网关/保障层产出的 decision_log、熔断状态、水位、用量，反过来帮人理解和优化这个平台。普通单步推理仍走 M1→M2 老路径，不经编排层。

## 2. 编排层内部模块划分：双执行模式 + 共享底座

任务分两类性质，用两种执行模式（Anthropic《Building Effective Agents》的 Agent vs Workflow 区分）：

```
                  ┌──────────────────────────────────────────────┐
   /v1/agent ───► │  ReAct Agent     reason→act→observe（有界循环）  │ Tier 1 诊断（探索式）
                  │       │                                        │
 /v1/skills/* ──► │  Workflow Runner 声明式 skill 模板（确定步骤）   │ Tier 2 报表 / Tier 3 建议
                  │       └────────┬─────────┘                      │
                  │                ▼  共享底座                       │
                  │  ④ Tool Registry（MCP 形状，自写 + 外部适配口）    │
                  │  ⑥ Telemetry 遥测底座（窗口聚合，Tier 0）         │
                  │  ⑦ State/Checkpoint（Redis，状态机 + 断点续跑）    │
                  │  ① ModelClient（mock / 真实可换，经 M2 执行器）    │
                  └──────────────────────────────────────────────┘
```

| 组件 | 职责 |
|------|------|
| ReAct Agent | 探索式诊断：模型自主决定下一步调哪个工具，观测结果后继续，直至给出结论（有界） |
| Workflow Runner | 确定式流程：按 skill 模板的有序步骤执行（取数→分析→出报表），步骤间传上下文 |
| ④ Tool Registry | mini MCP：工具注册/发现/执行；schema 对齐真实 LLM tool 形状；预留外部 MCP client |
| ⑥ Telemetry | Tier 0 地基：把 trace 摘要按时间存 Redis，支持按窗口/维度聚合查询 |
| ⑦ State/Checkpoint | 任务状态机 + 节点/轨迹检查点；断点续跑（TD-4） |
| ① ModelClient | 模型调用抽象契约；mock/真实同接口；调用经 M2 SafeguardedExecutor |

> 设计原则：**两种模式各用其长，但底座（工具/遥测/状态/模型）只有一套。** ReAct 用于"不知道下一步该查什么"的诊断；Workflow 用于"步骤已知、要可复现可调度"的报表与建议。

## 3. 两条执行流水线

### 3.1 ReAct 诊断流（Tier 1）

```
提交诊断目标（如"为什么最近多 503"）
   │
   ▼
生成 task_id / trace_id；初始化 messages = [系统提示 + 目标 + 可用工具清单]
   │
   ▼  循环（最多 max_steps）：
   ├─ ModelClient.call(messages, tools) ──► M2 执行器 ──► 后端
   │     │
   │     ├─ 返回 tool_calls ─► Tool Registry 执行（只读查询）─► observation 追加进 messages
   │     │                      └─ 每步 (action, observation) 写轨迹检查点
   │     └─ 返回 content（最终结论）─► 结束，返回诊断结果
   │
   └─ 超过 max_steps 仍未收敛 ─► 任务失败（防不收敛）
```

### 3.2 Workflow 报表/建议流（Tier 2 / 3）

```
触发 skill（如 weekly_usage_report，可由调度触发）
   │
   ▼
按 skill 模板取下一步（步骤间按声明顺序，依赖已满足）
   │
   ├─ 该步已有检查点？──是──► 读结果跳过（幂等）
   │                    └否──► 执行：tool 步（取数/出报表）或 model 步（分析）
   │                             成功 ─► 写检查点；失败 ─► 任务转 partial，停止
   ▼（所有步完成）
任务成功；返回最终产物（报表 / 建议清单）
```

## 4. 核心组件详细设计

### 4.1 ① ModelClient —— 模型后端抽象（接真模型的关键）

**契约（对齐真实 LLM function-calling 形状）**
```
ModelRequest  : { messages:[{role,content}], tools:[ToolSchema], params }
ModelResponse : 二选一 → { content:str }  或  { tool_calls:[{name, arguments}] }
ModelClient（协议）: async call(backend, req: ModelRequest) -> ModelResponse
```
- `MockModelClient`（M3 现用）：脚本化返回 content 或 tool_calls，模拟模型决策（如"看到 503 关键词就调 query_traces"）。
- `AnthropicModelClient` / `OpenAIModelClient`（接真模型时新增）：调 provider SDK，映射到同一 `ModelResponse`；工具 schema 直接对应 provider 的 tools 字段。
- **调用路径**：ModelClient 的每次 call 都经 **M2 SafeguardedExecutor** 打后端，熔断/重试/降级对编排层透明。

**与 M2 的衔接（兼容垫片，控制改动面）**
- M2 执行器现签名 `call(backend, input_text)->dict` 泛化为收 `ModelRequest`、返回 `ModelResponse`；熔断/重试/降级逻辑不变（只关心成功/失败）。
- M1 简单 `/v1/infer` 仍走老语义：input 包成单条 user 消息、取 `content` 返回。**现有 70 测试保持绿**。这是触及既有代码的唯一处，受控。

### 4.2 ④ Tool Registry —— mini MCP（FR-2.2）

- **注册**：启动时注册工具 = `{ name, description, parameters(JSON schema), handler }`。
- **发现**：`GET /v1/tools` 列清单；schema 形状对齐真实 LLM tool 定义，可直接喂模型。
- **执行**：按 name 查表、校验参数、调 handler（async）。
- **M3 工具集（全只读，自写，读本平台数据）**：

| 工具 | 作用 | 数据源 |
|------|------|--------|
| `query_metrics(metric, group_by, window)` | 错误率/延迟分位/QPS/拒绝率，按 模型/后端/调用方/reason 维度 | 遥测底座(§4.3) |
| `query_traces(filter, window)` | 捞符合条件的 decision_log（如最近 503/被 preempted 的） | 遥测底座 |
| `get_backend_health()` | 各后端熔断状态/healthy/当前水位 | /health 同源（Redis + config） |
| `query_usage(group_by, window)` | 按调用方/部门/模型聚合调用次数、token | 遥测底座 |
| `get_config(section)` | 读当前 callers/backends/thresholds（Tier 3 建议要对照现状） | 内存配置 |
| `render_report(data, format)` | 把数据套成 markdown/表格报表 | 纯函数 |

- **外部 MCP 适配口（预留，不实现）**：Registry 设计成能容纳"来自外部 MCP server 的工具"。将来加一个真 MCP client，把 Prometheus/GitHub/Slack/Postgres 等 server 暴露的工具注册进**同一张表**，ReAct/Workflow 无感。M3 不写传输层。

### 4.3 ⑥ Telemetry 遥测底座（Tier 0，工具的数据地基）

**现状**：decision_log 在内存 TraceStore（最近 ~1000 条环形缓存），熔断/水位在 Redis，无按时间窗的聚合，无 token/用量记录。

**M3 要补的最小遥测层**：
- 每个请求完成时，把一条**摘要**（trace_id, ts, caller, model, backend, reason, total_ms, degraded/served_fallback, 估算 token）推入 Redis（按时间戳的 sorted set / stream）。
- 提供按 `window` 的聚合：count、error_rate、latency 分位、按维度 group_by。
- token 计数 M3 可用"按 input/output 长度估算 + 价格表"占位（真实 token 计数在接真模型后由 ModelResponse 提供）。

> 这层只为"让工具有真数据可读"，不追求完整时序库（那是 M4 的 Prometheus）。

### 4.4 ReAct Agent（Tier 1，招牌）+ 轨迹幂等

- **有界循环**：reason→act→observe，`max_steps` 封顶防不收敛。
- **模型决定调工具**（FR-2.2 核心）：靠 `ModelResponse` 返回 `tool_calls` 还是 `content` 驱动。
- **轨迹幂等（TD-4 在 ReAct 下的解法）**：
  - 每完成一步，把 `(action, observation)` 追加进轨迹并写检查点（Redis）。
  - 续跑时：用已记录的轨迹重建 messages、**不重新执行已跑过的工具**，从断点继续推理。
  - 双重安全：Tier 1 工具**全只读**，即使重执行也无副作用；真正贵的是 token，靠轨迹检查点省下。
  - 检查点粒度 = "一次工具调用"，与 Workflow 的步骤级一致。

### 4.5 Workflow Runner + skill（Tier 2）

- **skill = 一份声明式 workflow 模板**（命名 + 参数 + 有序步骤）。**注意：这是我们自定义的 runbook 概念，不是 Claude Skills**（沿用"skill"一词指代可复用的运维流程）。
- 步骤类型：`tool`（取数/出报表）、`model`（分析，走 ModelClient）。
- 步骤间传上下文：后续步骤可引用前序输出（约定 `{{step_id.output}}`，执行前解析）。
- **示例 skill `weekly_usage_report(window=7d)`**：
  ```
  s1 query_usage(group_by=caller, window)          # 取数
  s2 query_metrics(error_rate+latency_p99, window) # 取数
  s3 model: 分析 s1/s2，写出趋势、异常点、SLA 达成   # LLM 分析（走 M2）
  s4 render_report(markdown)                       # 出报表
  ```
- **可调度**：报表天然适合 cron 触发（如周一早八跑），与平台调度能力衔接。

### 4.6 优化建议（Tier 3，只读）

- 实现为**只读的 advisory skill**（Workflow 的特例：取数 → 模型据规则给建议 → 输出建议清单），**不写配置**。
- M3 落 1–2 个：
  - `weight_rebalance_advisory`：对比各后端失败率/延迟 → 建议降权高失败后端（**呼应 M1 故意推迟的动态调权 P1**）。
  - `ratelimit_advisory`：某调用方长期顶格但系统有余量 → 建议提额；某调用方制造抢占 → 建议降额。
- 输出是"建议 + 依据 + 预期影响"，由人决定是否手动改配置。Tier 4 自动落地留作后续。

### 4.7 ⑦ State / Checkpoint（FR-2.3）

**状态机**：`pending → running → (success | failed | partial)`。

**Redis key 规划（概念）**
| key 模式 | 存什么 | 说明 |
|----------|--------|------|
| `task:{id}:meta` | 状态、类型(agent/workflow)、创建时间、目标/skill 名 | 任务元信息 |
| `task:{id}:step:{step_id}` | Workflow 某步输出 | 步骤检查点 |
| `task:{id}:trajectory` | ReAct 的 (action, observation) 序列 | 轨迹检查点 |

**断点续跑**：执行前查检查点是否存在——存在则复用/重建、跳过执行；只补跑未完成部分。要么"成功并写检查点"要么"失败不写"，无半完成态 → 幂等干净。

## 5. 任务数据结构

| 字段 | 何时写入 | 用途 |
|------|----------|------|
| task_id | 提交时生成 | 唯一标识，续跑凭据 |
| trace_id | 提交时生成 | 全链路追踪，贯穿所有步骤/工具调用 |
| type | 提交时 | agent（ReAct）/ workflow |
| status | 状态变更 | 状态机当前态 |
| trajectory / step_results | 每步完成 | 检查点；中间结果 |
| decision_log | 每步追加 | 每次模型/工具调用的结果、耗时（与 M1/M2 同口径） |

## 6. 对外接口

| 接口 | 用途 | 关键行为 |
|------|------|----------|
| `POST /v1/agent` | 提交一次诊断（ReAct） | 给目标 → 跑有界循环 → 返回 task_id + 诊断结论 |
| `GET /v1/skills` | 列出可用 skill | 返回 skill 清单（名/参数/说明） |
| `POST /v1/skills/{name}/run` | 跑一个 skill（报表/建议） | 返回 task_id + 产物；同 task_id 重跑触发续跑 |
| `GET /v1/tools` | 列出可用工具 | name + description + schema |
| `GET /v1/tasks/{id}` | 查询任务 | 状态机态 + 中间结果 + decision_log |
| `POST /v1/infer` | （沿用 M1）单步推理 | 不经编排层 |

**错误语义**：400 目标/skill 非法或参数缺失；404 task_id/skill 不存在；200 成功/部分完成（body 内 status 区分）。

## 7. 关键设计决策与权衡

| 决策 | 选择 | 理由 / 放弃了什么 |
|------|------|-------------------|
| Agent 部署目的 | 平台内置 AIOps Copilot（只读） | dogfood M1/M2 telemetry，自含可演示，兑现 AIOps 定位；放弃"通用任务编排服务"作为首落地（引擎仍通用，留作后续） |
| 执行模式 | 双模式：ReAct + Workflow | 探索式诊断用 ReAct、确定式报表用 Workflow，各用其长；放弃单一模式（报表用 ReAct 不确定/费 token；诊断用固定流程丢探索力） |
| ReAct 幂等 | 轨迹检查点 + 只读工具 | 重放轨迹不重执行工具；Tier 1 工具只读，重执行也安全。化解"ReAct 难幂等"的顾虑 |
| 工具来源 | 自写 MCP 形状工具 + 留外部 MCP client 适配口 | 本平台 telemetry 无现成 MCP 懂；外部 server 待接真实 infra 再接，届时只加适配器 |
| skill 语义 | 自定义 runbook 模板（非 Claude Skills） | 可复用、可调度的运维流程；避免与 Claude Skills 概念混淆 |
| 写操作 | M3 不做（只读到 Tier 3 建议） | 避免护栏/审计复杂度；自动修复(Tier 4)风险高，单独阶段做 |
| 模型后端 | ModelClient 抽象，mock/真实同契约 | 编排不依赖 mock；接真模型只换适配器 |
| 打后端 | 复用 M2 保障执行器 | 不重复造熔断/重试/降级 |

## 8. M3 开发任务拆解（建议顺序，每步独立可验）

```
T1  ModelClient 抽象 + MockModelClient（脚本化 content/tool_calls）
    + 泛化 M2 执行器契约 + /v1/infer 兼容垫片（现有 70 测试保持绿）
T2  Tier 0 遥测底座：trace 摘要按时间入 Redis + 窗口/维度聚合查询
T3  Tool Registry（MCP 形状）+ 只读工具集 + GET /v1/tools（预留外部 MCP 适配口）
T4  State/Checkpoint + 任务状态机 + 续跑判定（共享底座）
T5  Workflow Runner（Tier 2）+ 「用量/SLA 报表」skill + render_report
    —— 报表端到端 + 中途失败续跑不重跑
T6  ReAct Agent（Tier 1，招牌）：有界 reason→act→observe + 轨迹检查点
    —— 注入后端熔断故障 → agent 调工具定位到根因；超界终止
T7  Tier 3 优化建议：weight_rebalance / ratelimit advisory（只读产出建议）
T8  API 封装：/v1/agent · /v1/skills(/run) · /v1/tools · /v1/tasks/{id}
    —— trace_id / decision_log 贯穿
T9  （可选）真实模型适配器 + 真 MCP client 适配口
```

> 顺序逻辑：先立模型契约（T1，定所有模型调用形状）与数据地基（T2），再搭工具（T3）与状态底座（T4）；然后先做确定式 Workflow（T5，简单可测）热身，再做招牌 ReAct（T6）；Tier 3 建议（T7）复用 Workflow；最后封接口（T8）。T9 接真模型/外部 MCP 独立可选，不阻塞。

## 9. M3 自测验证清单

- [ ] `MockModelClient` 按脚本返回 content / tool_calls；现有 M1/M2 共 70 项测试保持全绿。
- [ ] 发若干请求后，遥测底座能按窗口/维度聚合出 count / error_rate / latency。
- [ ] `GET /v1/tools` 返回工具清单；每个只读工具能读出真实遥测/配置。
- [ ] Workflow：跑「用量报表」skill → 取数→分析→出报表；中途某步失败 → partial；同 task_id 重跑命中检查点跳过已完成步骤（mock 计数验证未重复执行）。
- [ ] ReAct：注入"某后端连续 5xx 触发熔断"→ agent 调 query_traces/get_backend_health 等工具 → 结论指出根因（circuit_open / 5xx / served_fallback 兜底）。
- [ ] ReAct 超过 max_steps 未收敛 → 任务失败（不死循环）；轨迹续跑不重复执行已跑工具。
- [ ] Tier 3：构造"gpt-a 失败率高"的遥测 → `weight_rebalance_advisory` 产出"建议降权"+依据。
- [ ] 模型调用经 M2 保障层：后端故障时熔断/重试/降级仍生效（decision_log 可见）。
- [ ] `GET /v1/tasks/{id}` 返回状态机态 + 中间结果。
- [ ] （接真模型后）某后端指向真实 provider，诊断/报表端到端跑通，编排代码零改动。

## 10. 与后续阶段的衔接
