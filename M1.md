# M1 —— 网关层概要设计（HLD）

> 上游文档：《AI 推理调度平台 —— 产品需求文档 v1.1》
> 本阶段范围：Layer 1 网关接入层
> 对应需求：FR-1.1 鉴权 / FR-1.2 分层限流 / FR-1.3 模型路由 / FR-1.6 调用方分级与优先级调度
> 文档性质：概要设计（High-Level Design），描述模块怎么划、请求怎么流、数据怎么存、组件怎么协作。**不含代码**，但详细到可以直接对照开工。

---

## 0. 本阶段目标

一句话：**让一个带凭证的请求，能完整、安全、有序地穿过网关，最终被分发到一个合适的模型后端。**

M1 结束时应能演示：

1. 带合法凭证的请求被放行，非法的被拒（401）。
2. 机器调用方按服务限流、人类调用方按用户限流，超限被拒（429）。
3. 模拟容量紧张时，高优先级请求被保、低优先级被限。
4. 请求被路由到一个具体后端，且能看出路由到了哪个。

> M1 暂不接真实模型，后端用 **mock 后端**（能模拟成功/失败/超时/慢响应）。保障层（熔断/重试/降级）留到 M2，本期路由到后端后直接拿 mock 结果返回即可。

---

## 1. 网关在系统中的位置

```
   调用方 ──► [ 网关 Layer 1 ] ──► (M2 保障层) ──► 模型后端
                  │
                  └── 本期 M1 只做这一层，后端用 mock 顶替
```

网关是整个平台的**唯一入口**。所有横切关注点（鉴权、限流、调度、路由）都在这里收口，下层只管"执行被允许的请求"。

---

## 2. 网关内部模块划分

网关内部拆成 5 个职责单一的组件，按请求流经顺序排列：

```
                      ┌────────────────────────────────────┐
   请求 ──►           │            网关 Gateway              │
                      │                                      │
              ┌───────▼────────┐                             │
              │ ① 身份解析与鉴权 │  Authenticator              │
              └───────┬────────┘                             │
                      ▼                                      │
              ┌────────────────┐                             │
              │ ② 分层限流      │  RateLimiter                 │
              └───────┬────────┘                             │
                      ▼                                      │
              ┌────────────────┐                             │
              │ ③ 优先级调度    │  PriorityScheduler           │
              └───────┬────────┘                             │
                      ▼                                      │
              ┌────────────────┐                             │
              │ ④ 模型路由      │  Router                      │
              └───────┬────────┘                             │
                      ▼                                      │
              ┌────────────────┐                             │
              │ ⑤ 后端分发      │  Dispatcher（M1 接 mock）     │
              └────────────────┘                             │
                      │                                      │
              ┌───────▼────────┐  贯穿全程：上下文 RequestContext │
              │  配置中心 Config │  贯穿全程：可观测 Observability  │
              └────────────────┘                             │
                      └──────────────────────────────────────┘
```

各组件职责：

| 组件 | 职责 | 输入 | 输出 / 副作用 |
|------|------|------|---------------|
| ① Authenticator | 校验凭证、识别身份（机器/人类）、加载调用方画像 | 原始请求（含凭证） | 通过→注入调用方画像；失败→401 |
| ② RateLimiter | 按维度做令牌桶限流 | 调用方画像 | 通过→放行；超限→429 |
| ③ PriorityScheduler | 看全局容量水位，按优先级决定放行/限制 | 调用方优先级 + 当前水位 | 通过→放行；被抢占→429（标记原因） |
| ④ Router | 在健康后端中按策略选一个 | 请求模型名 + 后端列表 | 选中的后端标识 |
| ⑤ Dispatcher | 把请求发给选中的后端 | 后端标识 + 请求体 | 后端响应（M1 为 mock） |
| 配置中心 | 提供调用方画像、后端列表、各项阈值 | — | 被各组件读取 |
| 可观测 | 记录每步结果、耗时、决策 | 各组件埋点 | 日志 / 指标 |

> 设计原则：**每个组件是一道"闸门"，要么放行、要么拦截并给出明确原因。** 拦截必须发生在尽可能早的环节（鉴权 < 限流 < 调度 < 路由），避免把无效请求带到下游浪费资源。

## 3. 请求处理流水线（核心流程）

一个请求穿过网关的完整决策链：

```
请求进入
   │
   ▼
① 解析凭证 ──── 无凭证/非法 ──► 拒绝 401
   │ 合法
   ▼
  加载调用方画像（类型/优先级/配额/可用模型）
   │
   ▼
  校验：请求的模型是否在该调用方可用列表内 ── 否 ──► 拒绝 403
   │ 是
   ▼
② 分层限流（按 类型 选维度：机器→服务key / 人类→用户id）
   │             └── 超限 ──► 拒绝 429（原因：rate_limited）
   │ 通过
   ▼
③ 优先级调度：读全局容量水位
   │   ├─ 水位正常 ──────────────────► 放行
   │   └─ 水位紧张
   │        ├─ 高优先级 ─────────────► 放行
   │        └─ 低优先级 ──► 拒绝 429（原因：preempted）
   │ 通过
   ▼
④ 路由：在「该模型的健康后端」中按权重选一个
   │             └── 无可用后端 ──► 拒绝 503
   │ 选中
   ▼
⑤ 分发到后端（M1：mock 返回）
   │
   ▼
  记录用量 + 调用链（异步，不阻塞）
   │
   ▼
返回结果
```

**关键点：限流（②）和优先级调度（③）是两件事，不要混。**
- ② 管的是"**单个调用方**有没有超过它自己的配额"——保护的是公平性。
- ③ 管的是"**整个系统**容量够不够，不够时牺牲谁"——保护的是高优先级业务。
- 一个请求可能自己没超限（过②），但因系统整体紧张且自己是低优先级而被拒（卡③）。

## 4. 核心组件详细设计

### 4.1 ① Authenticator（鉴权与身份解析）

**要做的事**

1. 从请求中取出凭证（约定放在请求头，如 `Authorization`）。
2. 校验凭证有效性。
3. 用凭证查出**调用方画像（CallerProfile）**，注入请求上下文供后续组件使用。

**调用方画像（CallerProfile）—— 概念结构**

| 字段 | 含义 | 例 |
|------|------|-----|
| caller_id | 调用方唯一标识 | `svc:search` 或 `user:zhangsan` |
| type | 类型 | `machine` / `human` |
| owner | 归属 | 服务名 / 员工 / 部门 |
| priority | 优先级 | `high` / `low`（机器默认 high，人类默认 low） |
| rate_limit | 该调用方的限流阈值 | 每秒 N 次 |
| quota | 配额上限 | 每日 token / 调用次数 |
| allowed_models | 可用模型列表 | `[gpt, claude, local]` |

**设计决策**
- M1 阶段画像可以**先放配置文件 / 内存字典**，不必上数据库。凭证 → 画像 是一次快速查表。
- 鉴权必须无状态、快：不做远程调用，画像预加载到内存。
- 凭证格式上，机器用长随机串 API Key，人类可用代表员工身份的 Token（M1 可简化为也用 Key，但 type 标 human）。

**失败处理**：凭证缺失或无效 → 立刻 401，不继续。

---

### 4.2 ② RateLimiter（分层限流）

**要做的事**：判断该调用方在当前时间窗口内是否还有"额度"，有则放行并扣减，无则拒绝。

**算法选型：令牌桶（推荐）**
- 桶有容量上限（应对突发），按固定速率回填令牌。
- 每个请求消耗一个令牌；桶空则拒绝。
- 比固定窗口更平滑，比滑动窗口实现更简单。

**"分层"体现在限流的 key 维度不同**
- 机器调用方：限流 key = 服务标识（如 `ratelimit:svc:search`）。
- 人类调用方：限流 key = 用户/部门标识（如 `ratelimit:user:zhangsan`）。
- 组件根据画像的 `type` 自动选用哪个维度，对外是同一套逻辑。

**为什么状态必须放 Redis**
- 平台是多实例部署的，限流必须是**全局**的——同一个调用方打到不同实例，额度要算在一起。
- 令牌桶的"取令牌 + 回填 + 扣减"是**复合操作**，多实例并发下必须**原子**完成，否则会超卖（两个实例同时判断"还有 1 个令牌"都放行）。
- 因此这套逻辑要作为一个原子单元在 Redis 侧执行（这是 TD-1，面试重点）。

**Redis key 规划（概念）**

| key 模式 | 存什么 | 说明 |
|----------|--------|------|
| `ratelimit:{dimension}:{id}` | 当前令牌数 + 上次回填时间 | dimension ∈ svc/user；按调用方隔离 |

**失败处理**：桶空 → 429，响应头带 `Retry-After`（根据回填速率估算多久后有令牌）。

---

### 4.3 ③ PriorityScheduler（优先级调度）⭐ 本期亮点

**要解决的问题**：当后端容量快被打满时，不能让低优先级的员工实验流量把线上业务的资源挤掉。

**核心概念：全局容量水位（Water Level）**
- 用一个全局指标衡量系统当前繁忙程度。M1 可用**在途请求数**（当前正在被处理、还没返回的请求总数）作为水位代理指标——简单且够用。
- 设两条线：
  - **警戒线（high watermark）**：在途数超过它 → 进入"紧张"状态。
  - **解除线（low watermark）**：在途数回落到它以下 → 恢复"正常"状态。
  - 两条线留间隔（迟滞），避免在临界点反复抖动。

**调度逻辑**

| 系统状态 | 高优先级请求 | 低优先级请求 |
|----------|--------------|--------------|
| 正常（水位 < 警戒线） | 放行 | 放行 |
| 紧张（水位 ≥ 警戒线） | 放行 | 拒绝（429，原因 `preempted`）|

> M1 先做最简单的"紧张时直接拒绝低优先级"。排队 / 降速是 P2 增强，先不做，但设计上要给"被抢占"留一个独立的拒绝原因码，便于区分和观测。

**在途计数怎么维护**
- 请求进入 ⑤ 分发前 +1，请求结束（无论成功失败）-1。
- 这个计数也要放 Redis（多实例全局可见），同样需要原子增减。
- 注意：**必须保证请求结束一定 -1**，否则计数泄漏会让水位虚高、误判紧张。要用"无论如何都执行的收尾逻辑"兜底。

**Redis key 规划（概念）**

| key 模式 | 存什么 | 说明 |
|----------|--------|------|
| `inflight:global` | 全局在途请求数 | 进入 +1 / 结束 -1，需原子 |
| `inflight:backend:{name}` | 单后端在途数（可选） | 为后续按后端水位调度预留 |

**面试要点（TD-7）**
- 水位指标怎么选：在途数 vs QPS vs 后端延迟。在途数最直接反映"系统手上压了多少活"。
- 迟滞（两条水位线）为什么需要：防抖。
- 低优先级是拒绝还是排队：本期拒绝（实现简单、效果明确），排队是增强。
- 调度判定本身不能成为瓶颈：判定只读一个全局计数 + 比大小，开销极小。

---

### 4.4 ④ Router（模型路由）

**要做的事**：在"请求指定模型"对应的、且**健康**的后端集合里，按策略选一个。

**后端配置（概念结构）**

| 字段 | 含义 |
|------|------|
| name | 后端标识 |
| model | 它服务哪个模型 |
| address | 它的地址（M1 指向 mock） |
| weight | 路由权重 |
| healthy | 健康状态（M1 可先全为 true，健康检查在 M3/M2 完善） |

**路由策略（M1 实现）**
- 按模型名筛出候选后端。
- 在候选中做**加权轮询**：权重高的被选中概率大。
- `P1`：结合后端实时负载/健康动态调权——M1 先用静态权重，留好接口。

**失败处理**：该模型没有任何健康后端 → 503。

> M1 阶段后端是 mock，Router 选中后交给 Dispatcher，Dispatcher 直接返回 mock 构造的响应即可。真实分发、超时、保障逻辑在 M2。

---

### 4.5 配置中心 & 可观测（贯穿组件）

**配置中心（M1 极简版）**
- 一份配置，描述：调用方画像表、后端列表、各项阈值（限流速率、水位线、权重）。
- M1 用配置文件（如 YAML）+ 启动时加载到内存即可，不必做动态热更新（热更新留作增强）。
- 满足 NFR-6：改阈值改配置，不改代码。

**可观测（M1 最小集）**
- 每个请求贯穿一个 `trace_id`（完整链路追踪在 M4，M1 先生成并打日志）。
- 每道闸门记录：是否放行、拒绝原因、耗时。
- 关键指标先用日志承载，Prometheus 接入留到 M4。

## 5. 请求上下文（RequestContext）

一个贯穿所有组件的"随行包"，避免组件间反复传参、也方便观测。概念字段：

| 字段 | 何时写入 | 用途 |
|------|----------|------|
| trace_id | 入口生成 | 全链路追踪 |
| caller_profile | 鉴权后 | 后续组件读取身份/优先级 |
| requested_model | 入口解析 | 路由依据 |
| chosen_backend | 路由后 | 观测/分发 |
| decision_log | 每道闸门追加 | 记录每步放行/拒绝及原因 |
| timings | 每段记录 | 各组件耗时 |

## 6. 对外接口（语义约定，不含实现）

M1 只需暴露最小接口：

| 接口 | 用途 | 关键行为 |
|------|------|----------|
| 推理接口（同步） | 发起一次模型请求 | 走完整流水线 ①→⑤，返回 mock 结果 |
| 健康检查接口 | 查看网关存活与后端状态 | 返回各后端 healthy 状态、当前水位 |
| （内部）用量/决策查询 | 调试用，看某请求的 decision_log | 便于自测验证流水线行为 |

**统一拒绝原因码（M1 需要的）**

| HTTP | reason | 含义 | 来自组件 |
|------|--------|------|----------|
| 401 | `unauthenticated` | 凭证无效 | ① |
| 403 | `model_not_allowed` | 请求了无权访问的模型 | ① |
| 429 | `rate_limited` | 自身超过限流阈值 | ② |
| 429 | `preempted` | 系统紧张，低优先级被抢占 | ③ |
| 503 | `no_backend` | 无可用后端 | ④ |

> `rate_limited` 与 `preempted` 都是 429 但 reason 不同——这是有意为之，验收和观测时要能区分二者（验收标准里专门有这一条）。

## 7. 关键设计决策与权衡（记录下来，面试能讲）

| 决策 | 选择 | 理由 / 放弃了什么 |
|------|------|-------------------|
| 限流算法 | 令牌桶 | 平滑应对突发；放弃固定窗口（临界点突刺）和滑动窗口（实现更重） |
| 限流与调度分离 | 拆成两个独立闸门 | 职责清晰：限流保公平、调度保优先级；混在一起会难以解释和观测 |
| 水位指标 | 在途请求数 | 最直接反映系统负载；M1 不引入延迟统计的复杂度 |
| 低优先级处理 | 紧张时直接拒绝 | 实现简单、效果明确；排队/降速作为 P2 增强 |
| 限流/水位状态 | 放 Redis | 多实例全局一致；代价是每请求有 Redis 往返，靠原子操作和精简逻辑控制开销 |
| 后端 | M1 用 mock | 解耦真实模型成本与限流，专注网关逻辑；也方便后续 M2 模拟各种故障 |
| 配置 | 文件 + 内存 | M1 够用；动态热更新是后续增强 |

## 8. M1 开发任务拆解（建议顺序）

每个任务都应可独立验证，做完一个能看到效果再做下一个：

```
T1  搭起网关骨架 + 一个能收请求、原样返回的推理接口（跑通空管道）
T2  接入配置中心：定义调用方画像表、后端列表、阈值，加载到内存
T3  ① Authenticator：凭证校验 + 画像注入 + 模型权限校验（401/403）
T4  接 Redis；实现 ② RateLimiter 令牌桶，机器/人类双维度（429 rate_limited）
T5  ③ PriorityScheduler：在途计数 + 水位判定 + 优先级抢占（429 preempted）
T6  ④ Router：加权轮询选后端（503 no_backend）
T7  ⑤ Dispatcher：接 mock 后端，按选中后端返回模拟结果
T8  贯穿：RequestContext + trace_id + decision_log + 各段耗时日志
T9  健康检查接口 + 决策查询接口（自测用）
T10 写 mock 后端（能模拟成功/失败/超时/慢响应，为 M2 铺路）
```

> 顺序逻辑：先把空管道跑通（T1-T2），再从入口往出口一道闸门一道闸门加（T3→T7），最后补观测和辅助接口（T8-T10）。任何时候都有一个"能跑的版本"。

## 9. M1 自测验证清单

做完后逐条验证（对应需求文档第 11 节验收标准的 M1 相关项）：

- [ ] 无凭证 / 错凭证 → 401 `unauthenticated`。
- [ ] 合法凭证 + 请求无权模型 → 403 `model_not_allowed`。
- [ ] 机器调用方高频请求 → 按服务维度限流，超限 429 `rate_limited`。
- [ ] 人类调用方高频请求 → 按用户维度限流，超限 429 `rate_limited`。
- [ ] 制造高在途（如 mock 后端故意慢）压过警戒线：
  - [ ] 高优先级请求仍放行。
  - [ ] 低优先级请求被拒 429 `preempted`（且 reason 与 rate_limited 可区分）。
  - [ ] 在途回落到解除线以下后，低优先级恢复放行。
- [ ] 配了多个同模型后端时，按权重分布被路由（多次请求统计命中比例近似权重）。
- [ ] 某模型无可用后端 → 503 `no_backend`。
- [ ] 每个请求日志里能看到完整 decision_log（经过哪些闸门、结果、各段耗时）。
- [ ] 多实例部署（起两个网关进程）时，限流和水位是全局一致的（同一调用方打两个实例，额度合并计算）。

> 最后一条（多实例全局一致）很关键——它证明状态确实在 Redis 而非进程内存，是"工业级"的硬证据，面试可重点讲。

## 10. 与后续阶段的衔接
