# AI 推理调度平台 —— 产品需求文档（PRD）

> 版本：v1.1
> 性质：个人练手项目 / 简历项目
> 技术栈：Python（FastAPI + asyncio）、Redis、Prometheus（可选）
> 文档目的：在写任何代码之前，先把"做什么、为什么做、做到什么程度"定义清楚，作为后续开发与面试讲解的唯一依据。

---

## 0. 文档说明

本文档是产品需求文档（PRD），描述**要构建什么**和**为什么**，不描述**具体怎么用代码实现**。阅读对象是开发者本人（也是这个项目唯一的工程师）。

每一条功能需求都标注了**优先级**：

- `P0` —— MVP 必须有，没有它项目讲不通
- `P1` —— 让项目"显得工业级"的关键加分项
- `P2` —— 有时间再做，锦上添花

## 1. 背景与问题定义

### 1.1 背景

在企业内部，大模型（LLM）是一种被广泛复用的基础能力，调用方分为两类：一类是**线上服务**（搜索、推荐、客服等程序触发的高并发流量），另一类是**公司员工**（通过内部 AI 工具、个人研发助手等交互式使用）。当这两类调用方混在一起、规模从"一个人"扩大到"上百个服务 + 大量员工、每秒成千上万次请求"时，直连模型会暴露一系列工程问题，且两类调用方对系统的诉求并不相同。

### 1.2 要解决的核心问题

| 编号 | 问题 | 直连模式下的后果 |
|------|------|------------------|
| PB-1 | 调用方无序直连，缺乏统一入口 | 无法限流、计费、鉴权；一个业务爆量拖垮所有人 |
| PB-2 | 复杂任务需要多步模型调用与工具调用 | 编排逻辑散落在各业务方，重复且不可控 |
| PB-3 | 模型后端会超时、会失败、会限流 | 单点故障直接传导到线上业务，可用性无保障 |
| PB-4 | 缺乏可观测性 | 出问题无法定位，成本无法核算，性能无法优化 |

### 1.3 一句话定位

> 一个面向线上服务的 **LLM 推理中间层**：统一接入大模型请求，提供限流、多模型路由、Agent 编排、熔断降级与全链路可观测能力，在高并发下保障稳定的服务等级（SLA）。

---

## 2. 目标与非目标

### 2.1 目标（In Scope）

- G1：所有 LLM 请求经由统一网关接入，具备鉴权、限流、路由、计费能力。
- G2：支持多步任务的 Agent 编排，包含工具调用与中间状态管理。
- G3：在模型后端不稳定时，平台具备熔断、重试、降级能力，不让故障穿透到调用方。
- G4：具备全链路追踪与关键指标可观测，能产出压测报告证明系统能力。

### 2.2 非目标（Out of Scope）

明确**不做**的事，避免范围蔓延：

- 不自研大模型，也不做模型训练/微调。后端模型一律视为外部依赖（可用真实 API 或本地 mock）。
- 不做面向终端用户的聊天 UI。本项目是基础设施，调用方是程序而非真人。
- 不做完整的向量数据库实现。如需检索能力，调用现成向量库或用内存版 mock 占位。
- 不追求多语言支持、不做国际化。
- **不做"差异化全链路策略"**（即两类调用方走完全不同的路由/降级/超时/重试参数组合）。这是有意识划出的边界：本版本只在**接入层**区分调用方身份并做优先级调度，下层策略两类共用。差异化全链路作为未来可扩展方向（P2），现阶段刻意不做，避免范围发散、每块都不深。

## 3. 用户与角色

调用方（Caller）分为两类，二者对系统的诉求差异很大，是后续"分级调度"设计的依据：

| 角色 | 类型 | 说明 | 特点 | 核心诉求 |
|------|------|------|------|----------|
| **机器调用方** | M2M | 线上服务（搜索/推荐/客服）、其他 Agent | 高 QPS、程序触发、不能等 | 限流、低延迟、熔断容灾、稳定 SLA |
| **人类调用方** | H2M | 公司员工通过内部 AI 工具 / 个人研发助手使用 | QPS 低、交互式、容忍稍慢但重体验 | 鉴权到人、按人/部门配额、流式体验、用量可查 |

其他角色：

| 角色 | 说明 | 关心什么 |
|------|------|----------|
| **平台运维（Operator）** | 维护本平台的人（即开发者本人） | 系统是否健康、限流是否生效、熔断是否触发、成本多少 |
| **模型后端（Backend）** | 外部 LLM 服务（OpenAI / Claude / 本地模型等） | 不被打垮；故障时能被快速隔离 |

> 关键认知：本系统服务的是**程序（机器调用方）和员工（人类调用方）两类混合流量**。机器流量决定了系统必须面向高并发、低延迟、高可用；人类流量则要求鉴权到人、按人配额、注重交互体验。当后端容量紧张时，要保证机器调用方（线上业务）的优先级高于人类调用方（员工实验性调用）——这是系统的核心取舍。

## 4. 系统总览

### 4.1 四层架构

```
                调用方（线上服务 / Agent / 内部工具）
                          │
        ┌─────────────────▼─────────────────┐
        │   Layer 1  Gateway 接入层           │
        │   鉴权 · 限流 · 路由 · 计费 · 流式转发  │
        └─────────────────┬─────────────────┘
                          │
        ┌─────────────────▼─────────────────┐
        │   Layer 2  Agent 编排层             │
        │   任务拆解 · 多步调度 · 工具调用(MCP)   │
        └─────────────────┬─────────────────┘
                          │
        ┌─────────────────▼─────────────────┐
        │   Layer 3  治理层                   │
        │   服务发现 · 热点探测 · 调用链追踪      │
        └─────────────────┬─────────────────┘
                          │
        ┌─────────────────▼─────────────────┐
        │   Layer 4  保障层（贯穿全栈）          │
        │   熔断 · 重试 · 降级 · 状态持久化 · SLA  │
        └───────────────────────────────────┘
```

### 4.2 一次请求的生命周期

1. 调用方携带 API Key 发起请求 → 网关鉴权。
2. 网关检查该调用方的配额与限流 → 通过则放行，超限则拒绝（返回 429）。
3. 网关根据策略选择一个模型后端（路由）。
4. 若是简单请求 → 直接转发给模型；若是多步任务 → 交给 Agent 编排层。
5. 编排层按任务计划依次调用模型 / 工具 / 检索，中间状态写入 Redis。
6. 全程由保障层兜底：任何后端失败触发重试 / 熔断 / 降级。
7. 结果（流式或非流式）返回调用方；调用链与计费数据异步落库。

## 5. 功能需求

### Layer 1 —— Gateway 接入层

#### FR-1.1 鉴权（P0）
- 每个调用方分配凭证（机器调用方用 API Key，人类调用方用用户身份 / Token），请求必须携带。
- 校验失败返回 401，且不进入后续任何逻辑。
- 鉴权能识别调用方的**身份维度**：是机器（服务）还是人（员工），人类调用方还可归属到部门。
- 每个凭证绑定一套配置：所属类型（机器/人类）、配额上限、限流阈值、优先级、可访问的模型列表。
- 鉴权应是无状态的快速校验，不成为性能瓶颈。

#### FR-1.2 分层限流（P0）
- 支持**多维度限流**：机器调用方按服务（Key）限流，人类调用方按用户 / 部门限流，两套独立阈值。
- 至少实现一种算法：令牌桶或滑动窗口（推荐令牌桶）。
- 限流状态存储于 Redis，保证多实例部署下限流全局一致。
- "读取配额 + 扣减"必须是原子操作（避免并发超卖）。
- 超过阈值返回 429，并在响应头给出重试建议（如 Retry-After）。
- 与 FR-1.6 的优先级调度联动：限流不仅看单个调用方阈值，还受全局容量与优先级影响。

#### FR-1.3 模型路由（P0）
- 支持配置多个模型后端。
- 路由策略至少支持：按模型名指定、加权轮询。
- `P1`：支持按后端实时负载 / 健康状态动态调权（与保障层联动）。
- 路由决策需可观测（能看出某次请求被分到了哪个后端）。

#### FR-1.4 流式转发（P1）
- 支持以 SSE（Server-Sent Events）方式把模型的流式输出实时转发给调用方。
- 调用方消费速度慢时，需有背压机制，避免内存被未消费数据撑爆。
- 流式连接中断时要能优雅清理资源。

#### FR-1.5 计费与用量统计（P1）
- 统计每个调用方的请求数、token 消耗、成本。
- 统计逻辑异步执行，绝不阻塞主请求链路。
- 用量数据可按**机器调用方（服务）**和**人类调用方（员工 / 部门）**两类维度分别聚合查询。

#### FR-1.6 调用方分级与优先级调度（P1）
> 本版本最核心的设计亮点：在接入层区分调用方类型并做优先级调度，对应 JD"大流量分布式场景下的高可用"。

- 每个调用方带有**优先级**属性。默认：机器调用方（线上业务）= 高优先级，人类调用方（员工实验性调用）= 低优先级（优先级可在配置中调整，不写死）。
- 平台维护一个**全局容量水位**概念（如某模型后端的并发上限 / 在途请求数）。
- 当容量充足时：所有调用方正常放行，仅各自受单方限流约束。
- 当容量紧张（接近水位）时：触发**优先级抢占**——优先保障高优先级请求放行，对低优先级请求主动限制（拒绝 / 排队 / 降速），避免员工实验流量挤占线上业务资源。
- 抢占行为需可观测：能看出某次低优先级请求是因"容量紧张被降级"而非"自身超限"被拒。
- `P2`（明确不在本期）：两类调用方走差异化的路由 / 降级 / 超时策略。本期下层策略两类共用。

---

### Layer 2 —— Agent 编排层

#### FR-2.1 任务编排（P1）
- 支持把一个"多步任务"建模为有向无环图（DAG），按依赖关系执行。
- 每个节点可以是：一次模型调用、一次工具调用、一次检索。
- 支持节点间传递上下文（前一步的输出作为后一步的输入）。

#### FR-2.2 工具调用（mini MCP）（P1）
- 实现一个最小化的工具调用协议（参考 MCP 思想）。
- 工具需可注册、可发现：平台维护一份可用工具清单。
- 模型决定调用某工具时，编排层负责实际执行并把结果回填。
- `P2`：工具调用支持并发执行（多个无依赖工具同时调）。

#### FR-2.3 任务状态管理（P1）
- 多步任务的中间状态持久化到 Redis。
- 支持断点续跑：任务中途失败重试时，已完成步骤不重复执行（幂等）。
- 每个任务有明确状态机：待执行 / 执行中 / 成功 / 失败 / 部分完成。

---

### Layer 3 —— 治理层

#### FR-3.1 调用链追踪（P1）
- 每个进入系统的请求分配唯一 trace_id，贯穿所有层。
- 能还原一次请求经过了哪些层、调用了哪些后端、各段耗时多少。
- `P2`：对接 OpenTelemetry 标准。

#### FR-3.2 热点探测（P2）
- 统计高频调用方、高频模型，识别热点。
- 热点信息可用于动态调整路由权重或触发预警。
- 基于滑动窗口做实时统计。

#### FR-3.3 健康检查与服务发现（P1）
- 平台定期探测各模型后端的健康状态。
- 不健康的后端自动从路由候选中摘除，恢复后自动重新纳入。

---

### Layer 4 —— 保障层（项目最核心的亮点）

#### FR-4.1 熔断器（P0）
- 对每个模型后端独立维护熔断状态。
- 实现标准三态机：
  - **闭合（Closed）**：正常放行；统计失败率。
  - **打开（Open）**：失败率超阈值，直接拒绝请求（快速失败），不再打后端。
  - **半开（Half-Open）**：冷却一段时间后放少量试探请求，成功则恢复闭合，失败则重新打开。
- 熔断触发与恢复需有日志和指标记录。

#### FR-4.2 重试与退避（P0）
- 后端调用失败时，按策略重试。
- 使用指数退避 + 随机抖动（jitter），避免重试风暴。
- 限制最大重试次数，避免无限重试放大故障。
- 只对可重试错误（超时、5xx、限流）重试，对不可重试错误（鉴权失败、参数错误）不重试。

#### FR-4.3 降级（P1）
- 主模型不可用时，按策略切换备用模型。
- 所有后端都不可用时，返回预设的兜底响应（而非直接报错崩溃）。
- 降级行为对调用方透明，但需在响应中标记"已降级"，便于排查。

#### FR-4.4 状态持久化与故障恢复（P1）
- 限流计数、熔断状态、任务中间态等关键数据持久化到 Redis。
- 平台实例重启后，能从 Redis 恢复状态，不丢关键数据。
- 关注数据一致性：并发更新场景下状态不能错乱。

## 6. 非功能需求

| 编号 | 类别 | 要求 |
|------|------|------|
| NFR-1 | 并发 | 单实例应能承载较高并发（目标：压测给出具体 QPS 数字） |
| NFR-2 | 延迟 | 平台自身引入的额外延迟应尽量小；关注 P99 而非平均值 |
| NFR-3 | 可用性 | 后端故障时平台不崩溃；通过熔断/降级保障对外可用 |
| NFR-4 | 异步化 | 所有同步 IO（落库、日志、计费）不得阻塞主请求链路 |
| NFR-5 | 可观测 | 关键指标（QPS、延迟、错误率、熔断次数）可被采集和查看 |
| NFR-6 | 可配置 | 限流阈值、路由策略、熔断参数等通过配置调整，无需改代码 |
| NFR-7 | 可水平扩展 | 多实例部署时，限流与熔断状态全局一致（依赖 Redis 共享状态） |

## 8. 对外接口约定（概念层）

> 仅定义接口的"语义契约"，不涉及具体实现。

| 接口 | 用途 | 关键行为 |
|------|------|----------|
| 推理接口（同步） | 调用方发起一次普通模型请求 | 鉴权→限流→路由→（保障层兜底）→返回结果 |
| 推理接口（流式） | 调用方发起流式请求 | 同上，但以 SSE 持续返回 token |
| 任务接口 | 调用方发起多步 Agent 任务 | 进入编排层，返回 task_id 与最终结果 |
| 任务查询接口 | 查询某个任务的执行状态 | 返回任务状态机当前状态与中间结果 |
| 用量查询接口 | 查询某调用方的用量与成本 | 按维度聚合返回 |
| 健康检查接口 | 运维查看平台与后端健康 | 返回各后端健康/熔断状态 |

**统一错误语义**：

- 401 鉴权失败
- 429 触发限流
- 503 后端全部不可用（已降级或兜底）
- 504 后端超时（重试耗尽后）

## 9. 关键技术难点（需提前想清楚，也是面试主战场）

| 编号 | 难点 | 要回答的问题 |
|------|------|--------------|
| TD-1 | 分布式限流的原子性 | 多实例下如何保证"读配额+扣减"不超卖？为什么要用 Redis 的原子操作 |
| TD-2 | 熔断器半开态 | 半开态存在的意义？阈值和冷却时间怎么定？ |
| TD-3 | 流式背压 | 调用方消费慢时怎么不撑爆内存？ |
| TD-4 | Agent 任务幂等 | 重试时如何保证已完成步骤不重复执行？ |
| TD-5 | asyncio 不阻塞 | 哪些操作必须异步化？同步 IO 混进事件循环会怎样？ |
| TD-6 | 重试风暴 | 为什么退避要加随机抖动？不加会怎样？ |
| TD-7 | 优先级抢占 | 容量紧张时如何判定"水位"？怎么保证高优先级不被低优先级挤占？低优先级是拒绝还是排队？抢占判定本身如何不成为新瓶颈？ |

## 10. 里程碑与交付路线

| 阶段 | 时间 | 交付内容 | 对应需求 |
|------|------|----------|----------|
| M1 | 第 1 周 | 网关骨架：鉴权（双维度）+ 分层限流 + 优先级调度 + 模型路由，能跑通一个请求 | FR-1.1 / 1.2 / 1.3 / 1.6 |
| M2 | 第 2 周 | 保障层：熔断 + 重试 + 降级 | FR-4.1 / 4.2 / 4.3 |
| M3 | 第 3 周 | Agent 编排 + 工具调用 + 任务状态 | FR-2.1 / 2.2 / 2.3 |
| M4 | 第 4 周 | 调用链追踪 + 健康检查 + 压测报告 + 文档 | FR-3.x / NFR-5 |

> 建议优先完成 M1 + M2（网关 + 保障层），这两块是工业界最核心、面试最常被深挖的部分，先打通能保证项目"讲得通"。

## 11. 验收标准

项目视为达标，当且仅当：

- [ ] 一个带 API Key 的请求能完整走通：鉴权 → 限流 → 路由 → 返回结果。
- [ ] 超过限流阈值的请求被正确拒绝（返回 429）。
- [ ] 鉴权能区分机器调用方与人类调用方，且各自按对应维度（服务 / 用户）限流。
- [ ] 模拟容量紧张时，高优先级（线上业务）请求被优先保障，低优先级（员工）请求被主动限制，且能区分"容量抢占被拒"与"自身超限被拒"。
- [ ] 模拟某个后端连续失败时，熔断器能打开并快速失败，冷却后能自动恢复。
- [ ] 主后端不可用时，能自动降级到备用后端或返回兜底响应。
- [ ] 多步任务能按 DAG 执行，中途失败重试不重复执行已完成步骤。
- [ ] 每个请求能追踪出完整调用链与各段耗时。
- [ ] 产出一份压测报告，包含 QPS、P99 延迟、熔断触发前后对比数据。
- [ ] 有一份 README，能让别人 5 分钟内理解项目并跑起来。

## 12. 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 范围过大，做不完 | 烂尾，无法交付 | 严格按优先级，先 P0 后 P1，P2 可砍 |
| 真实模型 API 有成本/限流 | 开发调试受阻 | 用本地 mock 后端模拟，能模拟成功/失败/超时/慢响应 |
| 过度设计，陷入细节 | 进度拖延 | 以"能讲清楚原理"为目标，而非"功能大而全" |
| 压测环境不真实 | 数据没说服力 | 用 locust 等工具规范压测，明确标注测试环境配置 |
