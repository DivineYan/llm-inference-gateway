## 0. 本阶段目标

一句话：**让一个已经被网关放行的请求，在后端不稳定时也能拿到一个合理的结果——要么换条路成功，要么优雅地兜底，绝不让后端故障直接穿透给调用方。**

M2 结束时应能演示：

1. 某后端连续失败 → 熔断器打开 → 后续请求**快速失败**（不再打这个后端），冷却后**自动半开试探**，探针成功则恢复。
2. 后端偶发超时/5xx → 按**指数退避 + 随机抖动**重试，重试耗尽才放弃；鉴权/参数类错误**不重试**。
3. 主模型整体不可用 → **降级**到备用模型；全都不可用 → 返回**预设兜底响应**，并在结果里标记"已降级"。
4. 起两个网关实例 → 一个实例把某后端熔断打开，另一个实例**立刻也快速失败**（熔断状态全局一致）。

> M2 仍不接真实模型，继续用 M1 的进程内 mock 后端，靠它的 `behavior`（success/failure/timeout/slow）注入故障来验证保障逻辑。

## 1. 保障层在系统中的位置

```
   调用方 ──► [ M1 网关 ①②③④ ] ──► [ M2 保障层 ] ──► 模型后端(mock)
                  鉴权/限流/调度/路由        熔断·重试·降级
```

M1 的四道闸门负责"**这个请求该不该进、该走哪个后端**"；M2 接在 ④路由 之后、真正打后端的 ⑤分发 环节，负责"**打后端这件事怎么打得稳、打不通怎么办**"。

**关键边界（沿用 M1 §10 留好的接口）**：
- **唯一对 M1 的改动**：Router 从"选一个后端"改为"按权重返回**有序候选列表**"（写进 `ctx.candidates`，`chosen_backend` 取列表首个保持兼容）。执行器据此依次试，是熔断/重试/降级组合的前提。流水线骨架与其余四道闸门不动。
- M1 的 Dispatcher 是独立一层 —— M2 把保障逻辑包在"打后端"周围，**不改流水线骨架**。
- M1 的 mock 后端四种行为，正是 M2 的故障注入工具。
- 决策链 `decision_log` / `trace_id` 口径不变，M2 新增的保障决策追加进同一条链路。

## 2. 保障层内部模块划分

保障层不是一道新"闸门"，而是把 M1 的 ⑤Dispatcher 升级成一个**保障执行器（SafeguardedExecutor）**，内部由三个职责单一的组件协作：

```
            ┌─────────────────────────────────────────────────┐
   路由结果  │            SafeguardedExecutor 保障执行器          │
  (候选后端) │                                                  │
            │   for 候选后端 in 候选列表:                         │
            │      ┌──────────────┐                            │
            │      │ ① CircuitBreaker│ 该后端熔断吗？Open→跳过     │
            │      └──────┬───────┘                            │
            │             ▼ 闭合/半开                            │
            │      ┌──────────────┐                            │
            │      │ ② Retrier     │ 打后端 + 退避重试           │
            │      └──────┬───────┘                            │
            │             ▼ 成功→返回 / 失败→记账给①、试下一个     │
            │   候选耗尽 ─►┌──────────────┐                      │
            │             │ ③ Degrader    │ 切备用模型 / 兜底响应  │
            │             └──────────────┘                      │
            └─────────────────────────────────────────────────┘
```

各组件职责：

| 组件 | 职责 | 输入 | 输出 / 副作用 |
|------|------|------|---------------|
| ① CircuitBreaker | 按后端独立维护三态机；放行前判定、调用后记账 | 后端名 + 调用结果 | allow/deny；状态写 Redis（全局） |
| ② Retrier | 失败时按指数退避+抖动重试；区分可重试/不可重试 | 一次后端调用 | 成功结果 / 最终失败（含原因） |
| ③ Degrader | 主模型不可用时切备用模型；全挂时返回兜底 | 当前模型 + 失败信号 | 备用后端集合 / 兜底响应（标记 degraded） |
| 保障执行器 | 编排上面三者：候选迭代 × 熔断 × 重试 × 降级 | ctx.chosen_backend + 候选集 | 最终响应 + 保障决策日志 |

> 设计原则：**熔断、重试、降级是三件正交的事，分开实现、组合使用。** 熔断管"这个后端还能不能碰"，重试管"这一次没打通要不要再试"，降级管"这条路彻底断了走哪条备路"。混在一起会难以解释和观测。

## 3. 分发处理流水线（M2 核心流程）

请求穿过 M1 四道闸门后，进入保障执行器：

```
路由产出：该模型的候选后端（按权重排好序）
   │
   ▼
进入保障执行器  ── 在途计数 +1（M1 的 InflightGuard 包住整段，含退避等待）
   │
   ▼
┌─► 取下一个候选后端 ──── 没有候选了 ──► ③ 降级
│      │
│      ▼
│   ① 熔断判定：该后端熔断状态？
│      ├─ Open（且未到冷却）─────► 跳过，回到取下一个候选
│      ├─ Half-Open（拿到探针名额）─► 放一个试探请求
│      └─ Closed ──────────────► 正常放行
│      │
│      ▼
│   ② 带重试地打后端：
│      调用 → 成功 ──► ① 记账(成功) ──► 返回结果 ✅
│             └─ 失败
│                ├─ 不可重试错误（鉴权/参数）─► 不重试，直接失败
│                └─ 可重试错误（超时/5xx/限流）
│                     ├─ 未达最大次数 ─► 退避(指数+抖动)后重试同后端
│                     └─ 达到最大次数 ─► 放弃这个后端
│      │
│      ▼
│   ① 记账(失败) ── 失败率超阈值/探针失败 ──► 把该后端熔断打开
└──────┘  回到取下一个候选

③ 降级：
   ├─ 该模型有备用模型？ ──► 用备用模型的候选集，回到上面的循环（只降一级，防止级联）
   └─ 没有备用 / 备用也挂 ──► 返回预设兜底响应（标记 degraded=true）
   │
   ▼
在途计数 -1（finally 兜底，无论成功失败）
   │
   ▼
返回结果（成功 / 降级结果 / 兜底）+ 写 decision_log
```

**关键点：三层"换路"的粒度不同，不要混。**
- **重试（②）**：同一个后端，针对**瞬时**故障（一次超时），原地再试。
- **候选迭代（执行器）**：同一个模型的**另一个**后端，针对某后端**持续**不行（熔断/重试耗尽）。
- **降级（③）**：换**另一个模型**，针对该模型**整体**不可用。

## 4. 核心组件详细设计

### 4.1 ① CircuitBreaker（熔断器）⭐ 本期亮点

**要解决的问题**：一个后端已经在持续失败时，继续把请求打过去只会让调用方一起陪着超时——既浪费时间又拖垮自己。熔断器让平台**快速失败**，把故障后端隔离开。

**标准三态机（FR-4.1）**

| 状态 | 含义 | 行为 | 转移条件 |
|------|------|------|----------|
| **Closed 闭合** | 正常 | 放行所有请求；统计失败率 | 失败率 ≥ 阈值（且样本足够）→ Open |
| **Open 打开** | 已隔离 | 直接拒绝（快速失败），**不打后端** | 经过冷却时间 → Half-Open |
| **Half-Open 半开** | 试探 | 只放**少量探针**请求 | 探针成功 → Closed；探针失败 → Open（重置冷却） |

**半开态为什么必须有（TD-2）**
- 没有半开态，要么永远开着（后端恢复了也不知道）、要么靠定时器盲目全量放行（后端没恢复会瞬间被打垮二次雪崩）。
- 半开是"**小成本试探**"：只放 1～N 个探针，用极小流量探测后端是否恢复，成功才全量恢复。
- 冷却时间 = 给后端喘息和自愈的窗口；阈值 = 容忍多大失败率才认定"坏了"。两者都进配置（NFR-6）。

**判定算法选型（失败率怎么算）—— 已定：滚动时间窗 + 失败率 + 最小样本数**

| 方案 | 怎么算 | 优点 | 缺点 |
|------|--------|------|------|
| **滚动时间窗 + 失败率（✓采用）** | 统计最近 W 秒内 调用数/失败数，失败率≥阈值 且 样本≥最小数 才跳闸 | 贴合 FR"失败率超阈值"；抗抖动 | 需窗口计数 + 最小样本保护 |
| 连续失败计数 | 连续失败 N 次跳闸 | 极简 | 50% 抖动型故障永远跳不了闸 |
| 滑动计数窗（最近 N 次） | 最近 N 次里失败占比 | 与流量速率解耦 | 每次调用要维护一个定长序列，Redis 侧偏重 |

> 最小样本数防止"1 次失败 = 100% 失败率"误判。窗口用 Lua 里"窗口起点到期则重置计数"的轻量实现，不维护定长序列。

**为什么状态必须放 Redis（FR-4.4 / NFR-7）**
- 多实例部署：实例 A 发现后端坏了跳了闸，实例 B 必须立刻也知道，否则 B 还在傻打。
- "读状态 + 记一次结果 + 可能触发状态转移"是**复合操作**，多实例并发下必须**原子**完成，否则状态会错乱（两个实例同时判定半开、各放一堆探针）。
- 因此熔断的"放行判定"和"结果记账"两个动作各做成一段 **Lua 脚本**在 Redis 侧原子执行（同 M1 的 TD-1 思路）。

**Redis key 规划（概念）**

| key 模式 | 存什么 | 说明 |
|----------|--------|------|
| `circuit:{backend}` | hash：state / 窗口内调用数 / 失败数 / 窗口起点 / opened_at / 半开探针名额 | 按后端隔离；状态机全部字段 |

**两个原子操作（Lua）**
- `allow(backend)` → 返回 `允许/拒绝` + 是否为探针。内部处理"Open 到点自动转 Half-Open""半开发放探针名额"。
- `record(backend, success)` → 更新窗口计数，按规则触发 Closed→Open / Half-Open→Closed/Open。

**失败处理**：Open 态下被拒的请求，不算"调用方的错"，交给执行器去试下一个候选 / 降级，不直接报错给调用方。

### 4.2 ② Retrier（重试与退避）

**要做的事**：一次后端调用失败时，判断"该不该再试"，该试就退避一会儿再试，限定最大次数。

**只对可重试错误重试（FR-4.2）**

| 错误类型 | 可重试？ | 理由 |
|----------|----------|------|
| 超时（timeout） | ✅ | 瞬时网络/负载抖动，再试可能就好 |
| 5xx 服务端错误 | ✅ | 后端临时故障 |
| 后端限流（429） | ✅（退避后） | 后端忙，等一下再试 |
| 鉴权失败 / 参数错误（4xx） | ❌ | 重试一万次也是同样的错，纯属浪费 |

> 这要求 mock 后端把失败**分类抛出**（超时异常 / 服务端错误 / 不可重试错误），Retrier 据此决策。M2 第一步就是给 mock 后端补这套**带类型的错误模型**。

**指数退避 + 随机抖动（TD-6）**
- 第 k 次重试等待：`min(base * 2^k, max_backoff)`，再叠加 `random()` 抖动。
- **为什么要抖动**：如果一批请求同时失败、又用同样的退避节奏，它们会**同时**重试，形成"重试风暴"——在后端最虚弱的时候给它一记同步的尖峰，二次打垮。抖动把重试时间打散，削平尖峰。
- **为什么限最大次数**：无限重试会放大故障、拖死调用方，且会持续给已经垂死的后端施压。达到上限就放弃这个后端，交给上层换路。

**退避等待与在途计数的关系（设计取舍 —— 已定：计入在途）**
- 请求在退避期间**仍然占着系统**（一个协程在等），M1 的在途水位语义上应把它算进去 → **整段保障执行（含退避）都在 InflightGuard 内**。
- 代价：大量重试会推高水位、可能触发 M1 的优先级抢占。这其实是**期望行为**（后端不稳时本就该收紧低优先级），但要明确写进设计，避免被当成 bug。

**失败处理**：重试耗尽仍失败 → 向执行器报"这个后端不行"（带最后一次错误类型），由执行器换下一个候选 / 降级 / 兜底，不直接对外报错。

### 4.3 ③ Degrader（降级）

**要做的事**：当某个模型**整体**没有可用后端（全部熔断 / 全部重试耗尽）时，不直接报错，而是换备用模型或返回兜底。

**两级降级（FR-4.3）**
1. **切备用模型**：配置里给模型配一个 `fallback_model`（如 `gpt → local`）。主模型候选全挂 → 用备用模型的候选集再走一遍执行流程。**只降一级**（备用的备用不再继续，防止级联绕圈）。
2. **兜底响应**：备用也不可用（或没配备用）→ 返回一个**预设的兜底文本**（如"服务繁忙，请稍后重试"），HTTP 503。

**降级必须对调用方透明但可观测**
- 结果体里加标记：`degraded: true` + `fallback_model`（用了哪个备用）/ `fallback_reason`。
- 调用方拿到的是一个**正常结构**的响应（不是异常崩溃），但能看出"这是降级来的"。
- decision_log 记录：主模型为什么挂、降到了哪、是否最终兜底。

**配置（概念）**

| 字段 | 含义 | 例 |
|------|------|-----|
| fallback_model | 某模型的备用模型 | `gpt → local` |
| fallback_response | 全挂时的兜底文本 | "服务繁忙…" |

**失败处理**：兜底是最后一道，永远有返回（哪怕是固定文本），保证"平台不崩溃"（NFR-3）。

---

### 4.4 错误模型与对外语义（贯穿）

**后端错误分类（M2 新增，mock 后端按此抛出，每个异常自带 `retryable`）**

| 错误 | 来自 mock 行为 | 可重试 | 重试耗尽后 |
|------|----------------|--------|-----------|
| BackendTimeout | timeout | ✅ | 该后端判失败 → 换候选 / 降级 / 兜底 |
| BackendServerError | failure（模拟 5xx） | ✅ | 该后端判失败 → 换候选 / 降级 / 兜底 |
| BackendBadRequest | bad_request（参数错） | ❌ | 不重试，立即判失败 → 换候选 / 降级 / 兜底 |

> （`BackendRateLimited`/后端 429 留作后续，本期未实现。）

**统一拒绝原因码（M2 新增，接续 M1 §6）**

| HTTP | reason | 含义 | 来自组件 |
|------|--------|------|----------|
| 200 | （正常）+ `degraded:true` `fallback_model` | 已降级到备用模型，仍成功返回 | ③ |
| 503 | `served_fallback` `degraded:true` | 全挂（含超时/失败重试耗尽、全部熔断），返回预设兜底响应 | ③ |

> **本期取舍**：遵循 FR-4.3「所有后端不可用时返回兜底响应而非报错」，总失败统一收敛为
> **503 `served_fallback`**（兜底永远有返回，NFR-3）。因此 M2 **不单独返回 504**——
> 超时重试耗尽也走同一条兜底路径，只是在 `decision_log`/`attempts` 里能看出失败类型。
> 沿用 PRD §8：503 = 后端全不可用（已降级或兜底）。

## 5. 配置中心增量（NFR-6）

M2 在 `config.yaml` 增加一个全局 `safeguard` 段（PRD §2.2 明确：下层策略两类调用方共用，所以是全局配置、不分调用方）：

```
safeguard:
  circuit:                  # 熔断参数（① CircuitBreaker）
    window_seconds:         # 失败率统计窗口
    failure_rate:           # 跳闸失败率阈值（0~1）
    min_samples:            # 最小样本数，不足不跳闸
    cooldown_seconds:       # Open → Half-Open 冷却时间
    half_open_probes:       # 半开态放行的探针数
  retry:                    # 重试参数（② Retrier）
    max_attempts:           # 单后端最大尝试次数
    base_backoff_ms:        # 退避基数
    max_backoff_ms:         # 退避上限
  degrade:                  # 降级参数（③ Degrader）
    fallback_model:         # 模型 → 备用模型映射
    fallback_response:      # 兜底文本
```

backend 配置沿用 M1（`behavior` / `delay_ms` 用于注入故障），M2 可加一个**后端调用超时阈值**字段供 Retrier 判超时。

## 6. RequestContext 增量

在 M1 的随行包基础上追加保障层字段：

| 字段 | 何时写入 | 用途 |
|------|----------|------|
| attempts | 每次后端调用后追加 | 记录试了哪些后端、各几次、结果、退避时长 |
| circuit_states | 每次判定时记 | 该请求看到的各后端熔断状态（观测） |
| degraded | 降级时 | true/false |
| fallback_model | 降级时 | 用了哪个备用模型 |

这些都进 `decision_log`，让"一次请求经历了哪些保障决策"可被 `/debug/trace` 完整还原。

## 7. 对外接口（增量）

接口形态不变（仍是 M1 的 `/v1/infer`、`/health`、`/debug/trace`），行为增强：

| 接口 | M2 变化 |
|------|---------|
| `/v1/infer` | 响应带 `degraded`（及降级时 `fallback_model`）标记；全挂兜底返回 503 `served_fallback` |
| `/health` | 新增展示各后端**熔断状态**（closed/open/half_open）+ opened_at |
| `/debug/trace/{id}` | decision_log 含保障决策（每次尝试、熔断状态、退避、降级） |

## 8. 关键设计决策与权衡（面试能讲）

| 决策 | 选择 | 理由 / 放弃了什么 |
|------|------|-------------------|
| 三组件分离 | 熔断/重试/降级各自独立、组合使用 | 职责清晰、可单独验证；混在一起难解释难观测 |
| 熔断判定 | 滚动时间窗 + 失败率 + 最小样本 | 贴合"失败率超阈值"、抗抖动；放弃连续失败（漏判抖动型故障） |
| 半开探针 | 限量探针（默认 1） | 小成本试探后端恢复；放弃定时全量放行（二次雪崩风险） |
| 重试退避 | 指数退避 + 随机抖动 | 抖动削平重试尖峰、防重试风暴（TD-6） |
| 重试错误分类 | 只重试瞬时/服务端错误 | 不可重试错误重试纯浪费且放大故障 |
| 熔断状态存储 | 放 Redis + Lua 原子 | 多实例全局一致（FR-4.4/NFR-7）；状态转移必须原子 |
| 降级层数 | 只降一级 + 兜底 | 防级联绕圈；保证最终一定有返回（NFR-3） |
| 退避期占在途 | 计入水位 | 语义正确（请求确实占着系统）；副作用是推高水位、收紧低优先级——视为期望行为 |
| 不做差异化策略 | 两类调用方共用保障参数 | 遵循 PRD §2.2 边界，避免范围发散 |

## 9. M2 开发任务拆解（建议顺序）

每个任务都应可独立验证（pytest 跑绿），做完一个再做下一个：

```
T1  错误模型：扩展 mock 后端按类型抛错（超时/5xx/不可重试）+ 错误分类函数
T2  ② Retrier：指数退避+抖动+最大次数+错误分类（只重试可重试错误）
T3  ① CircuitBreaker：三态机 + 两段 Lua（allow / record）+ 状态存 Redis
T4  ③ Degrader：备用模型切换 + 兜底响应 + degraded 标记
T5  保障执行器：编排 候选迭代 × 熔断 × 重试 × 降级，串成完整分发
T6  接入：用执行器替换 M1 Dispatcher，RequestContext/decision_log 增量，503 served_fallback
T7  /health 展示熔断状态；/debug/trace 展示保障决策
T8  多实例熔断状态全局一致（起两进程，一个跳闸另一个立刻快速失败）
```

> 顺序逻辑：先把"打后端会怎么失败"定义清楚（T1），再从内到外搭——重试（T2）和熔断（T3）是两块独立积木，降级（T4）依赖它们，执行器（T5）把三者编排起来，最后接入网关（T6）补观测（T7）和验证全局一致（T8）。

## 10. M2 自测验证清单

对应 PRD §11 验收标准的 M2 相关项：

- [ ] 某后端连续失败超过阈值 → 熔断打开，后续请求快速失败（不再打该后端，响应快）。
- [ ] 冷却时间到 → 半开，放探针；探针成功 → 闭合恢复；探针失败 → 重新打开。
- [ ] 后端偶发超时 → 按指数退避+抖动重试；重试耗尽后该后端判失败，进而换候选 / 降级 / 兜底。
- [ ] 不可重试错误（如参数错）→ 不重试，直接失败。
- [ ] 主模型全部后端不可用 → 自动降级到备用模型并成功返回，响应标记 `degraded:true`。
- [ ] 主备模型都不可用 → 返回预设兜底响应（503 `served_fallback`），平台不崩溃。
- [ ] `/health` 能看到各后端当前熔断状态；`/debug/trace` 能看到完整保障决策链。
- [ ] 起两个网关进程，一个实例触发某后端熔断 → 另一个实例对该后端也立刻快速失败（熔断状态全局一致）。

> 最后一条（熔断状态全局一致）和 M1 的限流全局一致是同一类硬证据：证明状态在 Redis 而非进程内存，面试可重点讲。

## 11. 与前后阶段的衔接

- **复用 M1**：直接用 M1 的 mock 后端（故障注入）、InflightGuard（在途计数）、RequestContext/decision_log（观测）、配置加载与 Redis 客户端（含 Lua 注册机制）。
- **M2 → M3 编排层**：Agent 多步任务的每一步模型/工具调用，都会复用 M2 的保障执行器来打后端——熔断/重试/降级对编排层透明。
- **M2 → M4**：熔断的开/关次数、重试次数、降级次数是 M4 可观测指标（NFR-5）的重要数据源；FR-3.3 健康检查会和熔断联动（探测结果直接驱动熔断状态）。
- **遗留增强**：FR-1.3 P1（按后端实时负载/健康动态调权）——熔断打开的后端从路由候选里摘除，可在 M2 接入后顺带打通（Router 查熔断状态过滤候选）。
